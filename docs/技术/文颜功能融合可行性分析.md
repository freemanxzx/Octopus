# Wenyan (文颜) 功能融合可行性分析报告（修订版）

> 制定日期：2026-04-09
> 修订说明：经过二次代码审计，发现了 doocs/md 的 **sync 浏览器扩展**多平台发布系统，修正了先前的部分结论。

---

## 一、核心结论

**文颜的全部优势功能都可以融合到 doocs/md 体系中，但需要区分"已内置"、"可直接移植"和"需要 Electron 桥接"三个层次。**

---

## 二、逐项功能审计

### 功能 1：外链转脚注

| 项目 | 详情 |
|------|------|
| **wenyan 做法** | 在 `@wenyan-md/core` 中拦截 `<a href>` 标签，将外部链接编号后移到文末 |
| **doocs/md 现状** | ✅ **已内置且更强**。`renderer-impl.ts` 第337行：`if (opts.citeStatus)` 触发 `addFootnote()`，自动去重并在文末生成 `引用链接` 区块 |
| **融合难度** | 🟢 零开发量。默认开启 `citeStatus` 即可 |
| **结论** | **无需移植，doocs/md 原生支持** |

### 功能 2：多平台适配与发布

| 项目 | 详情 |
|------|------|
| **wenyan 做法** | 在 `@wenyan-md/core` 中通过 `./publish` 和 `./wechat` 模块，配合 Swift 原生层实现微信草稿箱推送 |
| **doocs/md 现状** | ✅ **已内置且覆盖远超 wenyan**。通过配套的开源浏览器扩展 **sync**（`window.$sync`），已支持 **30+ 平台**的一键发布，包括：微信、知乎、头条、小红书、抖音、微博、B站、少数派、CSDN、掘金、博客园、Medium、简书、百家号、网易号、搜狐号、腾讯云、阿里云、华为云、豆瓣等 |
| **融合难度** | 🟢 零开发量（Web 版直接可用）；🟡 Electron 版需要将 sync 的逻辑内化到主进程中 |
| **结论** | **doocs/md 的多平台能力已远超 wenyan；Electron 版需将 Octopus MD 扩展逻辑整合进 Node 主进程** |

### 功能 3：微信图片直传 CDN（绕过浏览器跨域）

| 项目 | 详情 |
|------|------|
| **wenyan 做法** | Swift 层 `WechatAPI.swift` 直接调用微信 `add_material` API，支持 MD5 去重缓存、AccessToken 自动刷新 |
| **doocs/md 现状** | ❌ **不支持**。纯浏览器环境无法绕过微信 API 的 CORS 限制，只能依赖第三方图床（GitHub/阿里云OSS/七牛等13种） |
| **融合难度** | 🟡 中等。需在 Electron 主进程中用 Node.js 重写 `WechatAPI.swift` 的逻辑（约200行 Swift → 约150行 TypeScript），包括：`getAccessToken`、`uploadMaterial`（multipart/form-data）、`publishArticle`（draft/add）|
| **结论** | **需要 Electron 桥接才能实现。将 Swift 逻辑用 TypeScript 重写到 Electron 的 ipcMain 中** |

### 功能 4：本地文件系统读写（拖放 .md 文件打开）

| 项目 | 详情 |
|------|------|
| **wenyan 做法** | Swift 原生通过 `handleMarkdownFile` 读取本地 .md 文件内容并传递给前端 |
| **doocs/md 现状** | ⚠️ 部分支持。Web 版通过 `FileReader` API 支持文件拖放打开，但无法主动访问文件系统 |
| **融合难度** | 🟢 简单。Electron 的 `dialog.showOpenDialog` + `fs.readFile` 天然支持 |
| **结论** | **Electron 环境下自动获得，几乎零额外开发** |

### 功能 5：文章导出为长图

| 项目 | 详情 |
|------|------|
| **wenyan 做法** | 使用 `modern-screenshot` 库将渲染后的 DOM 转为 PNG，通过 Swift 保存到本地 |
| **doocs/md 现状** | ✅ **已内置**。使用 `html-to-image` 库实现类似功能 |
| **融合难度** | 🟢 零开发量 |
| **结论** | **无需移植** |

### 功能 6：Typora 主题导入

| 项目 | 详情 |
|------|------|
| **wenyan 做法** | 通过 `importCssHandler` 函数从 URL 拉取 CSS 并注册为自定义主题 |
| **doocs/md 现状** | ✅ **已内置且更强**。拥有完整的 `themeApplicator.ts` + `cssProcessor.ts` + PostCSS 管线，支持 CSS 变量注入、作用域隔离、自定义 CSS 覆盖 |
| **融合难度** | 🟢 零开发量 |
| **结论** | **无需移植，doocs/md 的主题系统更加成熟** |

### 功能 7：MCP 协议支持（AI Agent 自动发文）

| 项目 | 详情 |
|------|------|
| **wenyan 做法** | 有独立的 `wenyan-mcp` 仓库 |
| **doocs/md 现状** | ✅ **已内置**。`packages/mcp-server` 目录，已实现 MCP Server，可被 Claude/Cursor 等 AI 工具直接调用进行排版和发文 |
| **融合难度** | 🟢 零开发量 |
| **结论** | **无需移植** |

## 四、修正后的技术路线

### 之前结论的修正

> ⚠️ **重要修正**：先前报告低估了 doocs/md 的能力。经二次审计发现，doocs/md 通过 sync 浏览器扩展已经实现了比 wenyan 更广泛的多平台发布能力（30+ 平台 vs wenyan 仅 6 平台）。wenyan 的真正独特优势缩窄为：**微信原生 API 直连能力**（绕过跨域上传图片和推送草稿箱）。

### 确定的技术路线

```
Phase 1: 直接以 doocs/md 为基座 (零改造即获70%能力)
├── 外链转脚注 ✅ (开启 citeStatus)
├── 30+ 主题 ✅
├── 13种图床 ✅
├── MCP Server ✅
├── AI 辅助写作 ✅
└── 长图导出 ✅

Phase 2: 套 Electron 外壳 (获跨平台桌面客户端能力)
├── 本地文件系统完整读写
├── 系统菜单/托盘/快捷键
└── 离线使用支持

Phase 3: 移植 wenyan 独有的微信原生能力到 Electron 主进程
├── 用 TypeScript 重写 WechatAPI.swift (~150行)
│   ├── getAccessToken (带缓存)
│   ├── uploadMaterial (multipart 图片上传)
│   └── publishArticle (草稿箱推送)
└── 通过 ipcMain/ipcRenderer 桥接前端调用

Phase 4: 将 sync 浏览器扩展的多平台发布逻辑内化到 Electron
├── 不再依赖外部浏览器扩展
├── 可选引入 Playwright 实现无API平台的RPA发布
└── 构建数据闭环看板
```

### 最终结论

**wenyan 的所有优势功能都能被吸收，而且大部分在 doocs/md 中已经有同等或更强的实现。真正需要额外开发的只有一项：将微信原生 API 直连能力通过 Electron 的 Node 主进程重新实现（约150行 TypeScript）。这是一个非常明确且工作量可控的目标。**

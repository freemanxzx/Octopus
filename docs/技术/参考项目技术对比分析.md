# 双擎融合原理与产品架构深度分析报告

> 制定日期：2026-04-09
> 当前演进定调：**拥抱 Vue 3 生态，以 doocs/md 为内核，汲取 wenyan 跨平台特性与原生桥接理念，辅以 Electron + RPA 打造自动化客户端。**

基于前置对于代码库的细致走读，现提供全面且彻底的产品与技术对比分析。

---

## 一、doocs/md 核心架构全剖析

### 1.1 项目结构与技术底座
`doocs/md` 并非单体前端应用，而是一个高度模块化的 Monorepo。它包含了：
- **`packages/core`**：核心排版引擎。使用 `marked` v18 进行 Markdown AST 解析，配合 `highlight.js` (代码高亮)、`mermaid` (时序图渲染) 以及 `isomorphic-dompurify` (XSS 净化)。
- **`packages/shared`**：全局状态管理与类型层。
- **UI 呈现层 (`apps/*`)**：支持 `web` (传统浏览器使用)、`vscode` (编辑器插件形态) 和 `utools` (桌面快捷插件)。
- **构建栈**：Vue 3 + TypeScript + Vite + Pinia (状态中心)。

### 1.2 产品与技术优势
1. **渲染器 (Renderer) 的护城河极深**：它通过自定义的一套 PostCSS 插件（`postcss-custom-properties`, `postcss-calc`）动态结合主题 JSON，实现了数十套跨平台微信主题的零瑕疵渲染，甚至在转换过程中考虑了微信特殊的 `inline-styles` 重写方案。
2. **编辑器双向绑定成熟**：`apps/web/src/App.vue` 结合 `CodeMirror 6` 和 Pinia 进行了深度防抖优化与 DOM 映射。
3. **强大的图床/AI配置系统**：已有 13 种常见云图床的前端直连实现，及对外部大模型的接入能力。

### 1.3 核心劣势
1. **数据与能力受限于浏览器**：所有实现依赖浏览器的 LocalStorage，极易丢失数据。由于安全沙箱限制，无法调用原生系统级操作，并且所有对外（比如微信/知乎）的跨域请求都是重大的阻碍（依赖开发者手动配置 API 或浏览器代理）。
2. **排版后“半截子”流程**：排版结束后，用户只能点击“复制”然后人工去微信后台粘贴，并未打通**全自动化分发**最后一公里。

## 二、Wenyan (文颜) 核心架构全剖析

### 2.1 项目结构与技术底座
`wenyan` 走向了另一个极端，完全是一个深度绑定的 **Native 桌面应用程序**：
- **前端部分**：Svelte 5 + SvelteKit + Tailwind CSS。
- **原生部分**：完全基于 macOS 平台的 Swift 编写（在 `WenYan` 目录下）。
- **桥接部分 (Bridge)**：查看 `src/lib/bridge.ts` 可以发现，前端使用 `window.webkit.messageHandlers.wenyanBridge` 注册了一个巨大的回调池 (`invokeSwift`)，由 Swift 层执行真实逻辑。

### 2.2 产品与技术优势
1. **突破沙箱的客户端特权**：因为有 Swift 层托底，它可以轻易绕过浏览器的跨域拦截。`WechatAPI.swift` 可以无视 CORS 极速请求微信接口并返回结果给前端，实现了强大的 `UploadImage` （本地直传微信素材库且不耗费浏览器内存）与 `PublishArticleToDraft`。
2. **“链接转脚注”** 排版法则：由于微信公众号严禁外链（除白名单），wenyan在前端渲染拦截中，强制提取 `href` ，将外部链接顺序编号，附录在文章结尾。这是一个绝佳的产品优化细节。

### 2.3 核心劣势
1. **极致的平台锁定**：Swift 库加上 Svelte 构建，决定了它**不可能**编译在 Windows 或者 Linux 上运行，这就流失了大量的自媒体用户。
2. **渲染拓展性差**：Svelte 内核的生态不如 Vue 丰富，当尝试添加复杂的 PlantUML 或流程图渲染时，扩展性极其有限。

## 三、全面的演化与整合策略（How to do）

既然我们已经确定 **可以使用 Vue** 并且终局是要 **“打包成强大的自媒体客户端”**。结合以上的极细颗粒度分析，我们当前的最佳技术决策是：**“借 doocs/md 之骨，铸 wenyan 之翼，塑 RPA 之魂”**。

### 行动路线

#### Phase 1: 整合核心仓库 (The Base)
- 直接保留 `doocs/md` 的 **Vue 3 Monorepo 体系** 和其核心包 `@md/core`、`@md/shared`。不要重写为 React，这样直接白嫖它三年时间打磨的无懈可击的 `highlight` 解析与 30 种主题模板注入系统。
- 在我们的目录中，利用 `electron-vite` 创建一个 `apps/electron-client`。

#### Phase 2: 移植 Wenyan 桥接理念 (The Bridge)
- 废弃 Wenyan 的 `Swift + window.webkit` 逻辑。
- 在我们的 Electron 应用中，用 `ipcMain` 和 `ipcRenderer` 重构一套 `action.ts`（相当于我们拥有了跨平台、跨系统的原生特权），将 `doocs/md` 之前困在浏览器局限中的 HTTP 请求交由 Node.js 处理。
- 从 Wenyan 的 Markdown 处理模块中剥离其 **“外链转脚注”算法** 和 **API 数据组装格式**，以拦截器 (Interceptor) 的形式注入到 `doocs/md` 的渲染声明周期中。

#### Phase 3: RPA 补齐分发拼图 (The Evolution)
- 微信有开放 API，所以可以直接用 Node.js（Electron 主进程）代理文颜提供的 `publishArticleToDraft` 算法发送至草稿箱。
- 但为了完成我们的终极目标“多平台智能分发”，针对**知乎、小红书、掘金**，在 Electron 的 Main Process 中挂载 `Playwright` 隐藏容器。在底层渲染完（含脚注和图床的 HTML 后），由系统利用托管存放的 Cookie 自动加载后台进行富文本填充，实现免感发布。

---

> **总结：**
> 如果我们不用重新造一遍 React Markdown 排版和样式的轮子，采用 Vue 3 的确是**极其正确和省力**的路径。
> 结合了 `doocs/md` 万无一失的解析引擎，加上类似 `wenyan` 在桌面端的网络拦截和底层处理手法，再包裹上 Electron 的自动化，这将迅速蜕变为远超市面同类的智能化产品工具。我们将立即着手修改基础框架布局。

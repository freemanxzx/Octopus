# OctoPoster 工作流模式 vs 自主Agent模式 — 深度分析与选型决策

> 基于 Anthropic 官方 Agent 设计模式、业界 2026 最佳实践

> 本文档是对 `agent_technology_research.md` 的补充专题分析

## 一、两种模式的本质区别

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                    │
│   Workflow 模式 (工作流)              Agent 模式 (自主智能体)       │
│   ─────────────────                  ─────────────────              │
│                                                                    │
│   "代码编排 LLM"                     "LLM 编排自己"               │
│                                                                    │
│   代码定义路径:                       LLM 决定路径:                │
│   Step1 → Step2 → Step3              Observe → Plan → Act          │
│   ↓                                  ↓                             │
│   预设节点 + 条件分支                 动态选择工具 + 自主循环        │
│   ↓                                  ↓                             │
│   ✅ 可预测 / 可审计                  ✅ 灵活 / 能处理未知情况      │
│   ❌ 不能处理预料之外                 ❌ 不可预测 / 调试困难        │
│                                                                    │
└──────────────────────────────────────────────────────────────────┘
```

### 1.1 核心维度对比

| 维度 | Workflow (工作流) | Agent (自主智能体) |
|------|:-----------------:|:-----------------:|
| **决策者** | 开发者预编排 | LLM 实时决策 |
| **路径** | 固定 (确定性) | 动态 (概率性) |
| **可预测性** | ★★★★★ | ★★☆☆☆ |
| **灵活性** | ★★☆☆☆ | ★★★★★ |
| **速度** | 快 (直线执行) | 慢 (推理开销) |
| **成本** | 低 (最少LLM调用) | 高 (多轮推理) |
| **调试** | 简单 (步骤明确) | 困难 (路径不确定) |
| **适用** | 步骤明确/重复性任务 | 开放性/复杂推理 |

### 1.2 Anthropic 官方区分

Anthropic 在其官方 Agent 设计文档中明确区分：

> - **Workflow**："LLM 和工具按照**预定义的代码路径**被调度" — 代码是主人
> - **Agent**："LLM **动态地自主**决定它的过程和工具使用" — LLM 是主人

## 二、Anthropic 七大设计模式详解

Anthropic 总结了 7 种从简到繁的 Agentic 设计模式（前 5 种是 Workflow，后 2 种是 Agent）：

```
             简单 / 可控                     复杂 / 灵活
            ─────────────────────────────────────────────▶

  ┌─────────────────────────── WORKFLOW ──────────────────────────┐
  │                                                                │
  │  ①链式调用   ②路由分发   ③并行化   ④编排-工人   ⑤评估-优化   │
  │  Prompt      Routing     Parallel  Orchestrator  Evaluator    │
  │  Chaining                ization   -Workers      -Optimizer   │
  │                                                                │
  └────────────────────────────────────────────────────────────────┘
                                                          ▼
  ┌─────────────────────────── AGENT ────────────────────────────┐
  │                                                                │
  │           ⑥自主Agent              ⑦多Agent团队               │
  │           Autonomous Agent        Multi-Agent Teams           │
  │                                                                │
  └────────────────────────────────────────────────────────────────┘
```

### 各模式详解

#### ① 链式调用 (Prompt Chaining)

```
[输入] → [LLM步骤1] → [LLM步骤2] → [LLM步骤3] → [输出]
                 每步之间有确定性校验门
```

**特点**：最简单、最可控。每步输出是下步输入。可在步骤间插入确定性校验。
**OctoPoster 适用场景**：`主题 → 大纲 → 图片 → 文案`，这就是典型链式调用。

#### ② 路由分发 (Routing)

```
                  ┌── 类型A → 处理流A
[输入] → [分类器] ┼── 类型B → 处理流B
                  └── 类型C → 处理流C
```

**特点**：一个轻量分类器识别意图，路由到最合适的处理流。
**OctoPoster 适用场景**：用户输入 "帮我改图" vs "帮我写文案" vs "帮我做一套图文" → 路由到不同 Workflow。

#### ③ 并行化 (Parallelization)

```
              ┌── [子任务A] ──┐
[输入] ──────┼── [子任务B] ──┼── [聚合] → [输出]
              └── [子任务C] ──┘
```

**特点**：多个独立子任务同时执行，最后聚合结果。
**OctoPoster 适用场景**：8 页图片并发生成 (已有 SSE + goroutine 实现)。

#### ④ 编排-工人 (Orchestrator-Workers)

```
[目标] → [编排者 LLM] → 分解任务
              │
     ┌────────┼────────┐
     ▼        ▼        ▼
  [工人A]  [工人B]  [工人C]
     │        │        │
     └────────┼────────┘
              ▼
         [编排者汇总] → [输出]
```

**特点**：编排者负责理解目标、分解任务、分配给专家工人。
**OctoPoster 适用场景**："一句话生成一切" — 编排者分析要求，分配给 解析/创作/文案/审查 等工人。

#### ⑤ 评估-优化 (Evaluator-Optimizer)

```
[生成器 LLM] → [输出] → [评估器 LLM]
     ▲                        │
     │     未达标              │ 达标
     └────────────────────────┘  └──▶ 最终输出
```

**特点**：一个 LLM 生成，另一个 LLM 审查打分，循环直到满意。
**OctoPoster 适用场景**：图片质量审查 — 生成图片 → 评估器打分 → 低分重新生成。

#### ⑥ 自主 Agent (Autonomous Agent)

```
         ┌──────────────────────────────┐
         │                              │
         ▼                              │
  [感知环境] → [推理规划] → [选择工具]   │
                               │        │
                               ▼        │
                        [执行操作]       │
                               │        │
                               ▼        │
                        [评估结果] ──────┘
                               │
                           达到目标
                               │
                               ▼
                          [最终输出]
```

**特点**：LLM 自主决定每一步做什么，有反思循环，开放式。
**OctoPoster 适用场景**：对话式创作助手 — 用户模糊需求，Agent 自主探索最佳创作方案。

#### ⑦ 多 Agent 团队 (Multi-Agent Teams)

```
[用户需求] → [管理者 Agent]
                  │
        ┌─────── ┼ ───────┐
        ▼        ▼        ▼
    [解析Agent] [创作Agent] [文案Agent]
        │        │        │
        └─────── ┼ ───────┘
                 ▼
           [审查 Agent]
                 │
                 ▼
          [管理者汇总输出]
```

**特点**：多个专家 Agent 各司其职，通过 Handoff 协作。
**OctoPoster 适用场景**：复杂项目 — 品牌系列图文需要持续多轮迭代。

## 三、OctoPoster 场景逐一映射

### 3.1 场景-模式匹配分析

| # | OctoPoster 场景 | 路径确定性 | 推荐模式 | 理由 |
|---|----------------|:----------:|----------|------|
| 1 | **主题 → 大纲** | ★★★★★ | ① 链式调用 | 步骤固定：输入主题 → Prompt → 大纲 JSON |
| 2 | **URL → 提取 → 大纲** | ★★★★★ | ① 链式调用 | 步骤固定：抓取 → 清洗 → Prompt → 大纲 |
| 3 | **大纲 → N 张图片** | ★★★★☆ | ③ 并行化 | 每页独立生成，并行执行 |
| 4 | **主题 → 文案+标签** | ★★★★★ | ① 链式调用 | 一次 LLM 调用即可 |
| 5 | **图片去字/去背景/换背景** | ★★★★★ | ② 路由分发 | 分类用户意图 → 路由到对应处理 |
| 6 | **模板渲染** | ★★★★★ | ① 链式调用 | 确定性操作：选模板 → 注入数据 → Chromedp |
| 7 | **图片质量审查** | ★★★★☆ | ⑤ 评估-优化 | 生成 → 评分 → 不合格重试 |
| 8 | **"一句话生成一切"** | ★★☆☆☆ | ④ 编排-工人 | 需要自动分解任务、协调多个工人 |
| 9 | **对话式自由创作** | ★☆☆☆☆ | ⑥ 自主 Agent | 用户需求模糊，需要 Agent 自主探索 |
| 10 | **品牌系列图文项目** | ★☆☆☆☆ | ⑦ 多 Agent 团队 | 多轮迭代，角色协作 |

## 四、结论：Workflow-First + Agent-Enhanced 混合架构

### 4.1 黄金法则

> **"能用 Workflow 解决的，绝不用 Agent。"**
> 
> Agent 只在 Workflow 无法覆盖的**开放性、模糊性**场景才启用。
> — Anthropic 官方最佳实践

### 4.2 OctoPoster 推荐架构

```
┌────────────────────────────────────────────────────────────────┐
│                     OctoPoster 混合架构                          │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  入口层: 意图路由 (模式 ②)                               │    │
│  │                                                           │    │
│  │  用户输入 → 意图分类器 (轻量 LLM 或规则)                  │    │
│  │      │                                                    │    │
│  │      ├── "生成大纲"      → Workflow A                    │    │
│  │      ├── "生成图片"      → Workflow B                    │    │
│  │      ├── "写文案"        → Workflow C                    │    │
│  │      ├── "修图"          → Workflow D (路由子分类)        │    │
│  │      ├── "一句话全套"    → Workflow E (编排-工人)         │    │
│  │      └── "自由对话创作"  → Agent Mode                    │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  ┌─────────────────────── Workflow 区域 ──────────────────────┐ │
│  │  (覆盖 80% 场景 — 确定性路径)                               │ │
│  │                                                               │ │
│  │  Workflow A: 大纲生成 (链式调用 ①)                          │ │
│  │  [输入] → [提取/清洗] → [LLM 大纲推理] → [结构化校验] → 输出│ │
│  │                                                               │ │
│  │  Workflow B: 图片并发 (并行化 ③)                             │ │
│  │  [大纲] → [页1生成 ┬ 页2生成 ┬ ... ] → [质量评估⑤] → 输出  │ │
│  │                    └────────┘                                 │ │
│  │  Workflow C: 文案生成 (链式调用 ①)                          │ │
│  │  [主题+大纲] → [LLM 文案] → [标签提取] → 输出              │ │
│  │                                                               │ │
│  │  Workflow D: 智能修图 (路由 ②)                              │ │
│  │  [图片+指令] → [意图分类] → {去字/去背/换背/增强}→ 输出    │ │
│  │                                                               │ │
│  │  Workflow E: 一句话全套 (编排-工人 ④ + 评估-优化 ⑤)        │ │
│  │  [编排者] → 分解任务                                         │ │
│  │     ├─ [解析工人]: 提取内容                                  │ │
│  │     ├─ [创作工人]: 大纲+图片 (并行)                          │ │
│  │     ├─ [文案工人]: 标题+正文+标签                            │ │
│  │     └─ [审查工人]: 质量评估 → 不达标则循环修正               │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌─────────────────────── Agent 区域 ────────────────────────┐  │
│  │  (覆盖 20% 场景 — 开放性路径)                               │  │
│  │                                                               │  │
│  │  Agent Mode: 对话式创作助手 (自主 Agent ⑥)                  │  │
│  │  · 用户:"帮我做个护肤产品测评系列,要高级感"                  │  │
│  │  · Agent 自主: 分析需求→参考品牌库→选风格→试做→               │  │
│  │    征求反馈→调整→最终输出                                     │  │
│  │  · 工具: 可调用所有 Workflow 作为"子程序"                    │  │
│  │                                                               │  │
│  │  Team Mode: 品牌项目 (多 Agent 团队 ⑦) [未来]              │  │
│  │  · 策划Agent + 设计Agent + 文案Agent + 审核Agent            │  │
│  │  · 多轮迭代，跨日任务                                        │  │
│  └───────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────┘
```

### 4.3 为什么是 80% Workflow + 20% Agent

| 论点 | 说明 |
|------|------|
| **成本** | Workflow 每个任务 1-3 次 LLM 调用；Agent 可能 10-30 次。成本差 5-10 倍 |
| **速度** | Workflow 直线执行 3-5 秒；Agent 推理循环 15-60 秒 |
| **可靠性** | Workflow 可预测结果；Agent 每次输出可能不同 |
| **调试** | Workflow 步骤明确可追踪；Agent 需要额外 Tracing 基础设施 |
| **用户体验** | 80% 用户需求是明确的 (生成大纲/图片/文案)，Workflow 又快又稳 |
| **差异化** | 20% 的 Agent 能力 (对话式创作) 是竞品没有的杀手功能 |

### 4.4 Agent 作为 Workflow 的"升级版"

关键设计：**Agent 可以调用 Workflow 作为子程序**

```
自主 Agent 的工具箱:
├── run_workflow("outline_generation", {topic: "..."})   ← 调用 Workflow A
├── run_workflow("image_generation", {pages: [...]})     ← 调用 Workflow B
├── run_workflow("copywriting", {topic: "...", ...})     ← 调用 Workflow C
├── run_workflow("smart_retouch", {image: "...", ...})   ← 调用 Workflow D
├── run_workflow("full_pipeline", {input: "..."})        ← 调用 Workflow E
├── search_knowledge("爆款标题库")                        ← RAG 检索
├── ask_user("你倾向哪种风格？")                          ← 人工介入
└── save_result(images, copy, tags)                       ← 结果持久化
```

这样 Agent 不需要从零开始推理每个步骤，而是以 Workflow 为"能力单元"进行高层编排。

## 五、在 Dify 中的实现映射

| OctoPoster 模式 | Dify 实现方式 |
|-----------------|---------------|
| ① 链式调用 | Dify Workflow (LLM节点 → 条件节点 → 工具节点) |

```
Dify 应用类型                    OctoPoster 用途
```

## 六、实施建议

### 6.1 分阶段递进

| 阶段 | 模式 | 内容 | 工期 |
|:----:|:----:|------|:----:|
| **Phase 1** | Workflow Only | 5 个核心 Workflow (A-E) 覆盖 80% 场景 | 2 周 |
| **Phase 2** | + Agent 入口 | Dify Agent 应用 + Workflow 作为工具 | 1 周 |
| **Phase 3** | + 质量循环 | 评估-优化模式 (Workflow E 增强) | 1 周 |
| **Phase 4** | + Multi-Agent | LangGraph 多 Agent 团队 (高级用户) | 2 周 |

### 6.2 关键设计原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **Workflow-First** | 默认使用 Workflow，只在必要时升级为 Agent |
| 2 | **Agent 引用 Workflow** | Agent 不重复实现 Workflow 已有能力 |
| 3 | **确定性边界** | LLM 做推理和生成，代码做校验和执行 |
| 4 | **成本预算** | 每个 Agent 会话设 Token 上限和费用上限 |
| 5 | **渐进式披露** | 普通用户看到 Workflow UI，高级用户解锁 Agent 对话 |
